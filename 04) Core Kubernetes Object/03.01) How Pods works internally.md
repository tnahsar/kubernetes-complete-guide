# 📘 How Kubernetes Pod Object Works Internally

A Pod is the smallest deployable unit in Kubernetes.

A Pod usually contains:

* one or more containers
* shared networking
* shared storage

Pods run application workloads inside the Kubernetes cluster.

---

# Example Pod YAML

```yaml id="m2jlwm"
apiVersion: v1
kind: Pod

metadata:
  name: nginx-pod

spec:
  containers:
    - name: nginx-container
      image: nginx:latest

      ports:
        - containerPort: 80
```

---

# 🌐 Step-by-Step Internal Working of Pod

---

# 1️⃣ User Creates Pod Object

Command:

```bash id="n4mjlwm"
kubectl apply -f pod.yaml
```

The YAML is sent to:

```text id="q7m’wini"
kube-apiserver
```

---

# 2️⃣ kube-apiserver Validates and Stores Pod Object

Component:

```text id="c1m’wini"
kube-apiserver
```

Responsibilities:

* validates Pod YAML
* creates Pod object
* stores Pod state in etcd

At this stage:

```text id="w5m’wini"
Pod exists only as desired state metadata.
```

No container is running yet.

---

# 3️⃣ Scheduler Detects Unschedulable Pod

Component:

```text id="t8m’wini"
kube-scheduler
```

Responsibilities:

* watches for Pods without assigned nodes
* finds suitable worker node

Scheduler evaluates:

* CPU
* memory
* taints/tolerations
* affinity/anti-affinity
* resource availability

Example:

```text id="x2m’wini"
nginx-pod → worker-node-2
```

Scheduler updates Pod spec:

```yaml id="39rvfb"
nodeName: worker-node-2
```

---

# 4️⃣ kubelet Detects Assigned Pod

Component:

```text id="v9m’wini"
kubelet
```

(runs on every worker node)

Responsibilities:

* watches Pods assigned to its node
* ensures containers are running

kubelet notices:

```text id="k3m’wini"
"This Pod should run on my node."
```

---

# 5️⃣ kubelet Talks to Container Runtime

Component:

```text id="m6m’wini"
container runtime
```

Examples:

* containerd
* CRI-O
* Docker (older setups)

Responsibilities:

* pull container image
* create containers
* start containers

---

# 🌐 Container Creation Flow

```text id="u1m’wini"
kubelet
    ↓
container runtime
    ↓
pull nginx image
    ↓
create container
    ↓
start container
```

---

# 6️⃣ Pod Sandbox Is Created

Before containers start, Kubernetes creates:

```text id="r4m’wini"
Pod Sandbox
```

The sandbox provides:

* Pod network namespace
* Pod IP
* shared networking environment

Usually implemented using:

```text id="b8m’wini"
pause container
```

---

# 🌐 Why Pause Container Exists

The pause container holds:

* Pod network namespace
* Pod lifecycle namespace

All application containers join this shared namespace.

---

# 🌐 Pod Networking Model

```text id="y6m’wini"
Pod
 ├── pause container
 ├── nginx container
 └── sidecar container
```

All containers share:

* same Pod IP
* same localhost
* same port space

---

# 7️⃣ CNI Plugin Configures Networking

Component:

```text id="j5m’wini"
CNI Plugin
```

Examples:

* Calico
* Flannel
* Cilium

Responsibilities:

* assign Pod IP
* configure virtual interfaces
* connect Pod to cluster network

Example:

```text id="f2m’wini"
Pod IP → 10.244.0.5
```

---

# 8️⃣ Application Opens Ports

Inside the container:

* application process starts
* Linux kernel opens listening sockets

Example:

```text id="n7m’wini"
nginx listens on port 80
```

Important:

```text id="q1m’wini"
Applications open ports — not containers.
```

Ports belong to:

* Pod shared network namespace

---

# 🌐 Traffic Flow

```text id="s4m’wini"
Client Request
      ↓
Pod IP:80
      ↓
Linux Kernel
      ↓
nginx process inside container
```

---

# 9️⃣ Pod Status Updated

kubelet continuously updates Pod status:

Example:

```text id="z9m’wini"
Pending
Running
Succeeded
Failed
CrashLoopBackOff
```

Status stored through:

```text id="g3m’wini"
kube-apiserver → etcd
```

---

# 🌐 Final Running Architecture

```text id="p6m’wini"
Pod
 ├── Network Namespace
 ├── Storage Volumes
 ├── pause container
 └── Application Containers
```

---

# 📘 Important Clarification

Pod itself does NOT:

* directly run containers
* directly schedule itself

Pod is:

* a declarative Kubernetes object

Actual work is performed by:

* kube-scheduler
* kubelet
* container runtime
* CNI plugins

---

# 🌐 Kubernetes Components Involved

| Component         | Responsibility               |
| ----------------- | ---------------------------- |
| kube-apiserver    | Stores Pod object            |
| etcd              | Stores cluster state         |
| kube-scheduler    | Assigns Pod to node          |
| kubelet           | Manages Pod lifecycle        |
| container runtime | Runs containers              |
| CNI plugin        | Configures networking        |
| Linux kernel      | Manages namespaces & sockets |

---

# 📘 Pod Lifecycle Summary

```text id="d8m’wini"
User creates Pod
        ↓
kube-apiserver stores Pod
        ↓
Scheduler assigns node
        ↓
kubelet detects Pod
        ↓
Container runtime starts containers
        ↓
CNI configures networking
        ↓
Pod becomes Running
```

---

# 🔑 One-Line Interview Answer

> A Kubernetes Pod is the smallest deployable unit that runs one or more containers inside a shared network namespace, while kube-scheduler, kubelet, container runtime, and CNI plugins work together to run and manage the Pod.
